# 장애 대응 매뉴얼 (Incident Response Manual)

## 1. 장애 대응 기본 원칙
- 장애 발생 시 즉시 원인 파악 및 영향 범위 확인
- 상황창 또는 단체 메신저에 사실 중심의 진행 상황 공유
- 10분 이내 1차 보고, 이후 30분 단위로 중간 보고
- 추정 대신 확인된 사실만 보고, 원인은 추후 정리
- 복구 완료 후 재발 방지 대책 포함 최종 보고 작성

## 4. 장애 유형별 대응 지침

| 장애 유형 | 주요 증상 | 1차 조치 | 담당 부서 | 비고 |
|------------|------------|-----------|------------|------|
| 로그인 오류 | ERP, 웹포털 로그인 실패 | 인증 서버 재기동, 세션 초기화 | 운영팀 | 세션 폭주 여부 확인 |
| DB 접속 오류 | “DB Connection failed” | DB 재기동 후 Connection Pool 확인 | DB팀 | Connection 수 제한 확인 |
| 네트워크 지연 | 응답 지연, 타임아웃 발생 | 네트워크 트래픽 확인, 방화벽 점검 | NW팀 | 스위치 포트 상태 점검 |
| WAS 장애 | 500/502 에러 다수 발생 | 서비스 재기동, 로그 분석 | 서비스팀 | 로그 패턴 기반 원인분석 |
| 스토리지 용량 부족 | 파일 저장 실패 | 불필요 로그 삭제, 용량 증설 요청 | Infra팀 | Disk alert 모니터링 강화 |
| 배치 실패 | 스케줄 실행 중단 | Batch log 확인, 수동 재실행 | 배치팀 | 실행 계정 권한 확인 |

---

## 5. 보고 시 유의사항
- “복구 중” 시점에는 추정 원인 기입 금지
- “조치 완료” 전에는 재발 방지대책 미작성
- 기술 용어는 약어 대신 풀어쓰기 (예: WAS → Web Application Server)
- 외부 영향 가능성(전원, 네트워크 등)은 별도 섹션으로 기재

---

## 6. 예시 문장
- 현재 서비스 지연 현상이 확인되어 원인 분석 중입니다.
- DB Connection 오류로 인한 서비스 불가 현상이 발생하여 복구 작업을 진행 중입니다.
- 재기동 완료 후 서비스 정상화되었으며, 로그를 통해 원인을 추가 분석할 예정입니다.
- 일시적 세션 폭주로 인한 장애로 확인되었으며, 세션 타임아웃을 조정하였습니다.
